1. Introduction

The text input file formats of GEMIPM2K program, in general, correspond to three data structures that keep the definition of chemical system (DATACH structure) and work data of the GEM IPM algorithm (MULTI structure), with part of data that change from node to node exchanged through the DATABR ("data bridge recipe") structure. Accordingly, the input consists of three file types: the DCH, the IPM and the DBR. All these files have the same free-text data format. The files can be typed using any ASCII text editor, or easily produced from a SysEq record (calculated equilibrium state) in the GEM-Selektor package. To do the latter, simply open a SysEq record in "Single Thermodynamic System" dialog and re-calculate it. Then select a menu command "Data" "Write GEMIPM2K files ..." and follow the appearing dialogs.

The contents of DCH file, read only once at the initialization of the GEMIPM2K session, are copied into both DATACH and MULTI memory structures.

The IPM file contains the data that are irrelevant for the mass-transport part; it is read once after the DCH file, and the contents are sent into the MULTI memory structure. The IPM file contains numerical settings of the GEM IPM algorithm, as well as the parameters of non-ideal mixing models in phases-solutions, and/or properties of sorption phases and surface complexes belonging to them.

One or many DBR files contain the data for bulk composition, temperature, pressure, and metastability constraints (GEM input). At the same time, such files when written by GEM-Selektor or (coupled) GEMIPM2K code, contain the GEM equilibrium speciation results (output) such as the dual solution (vector u), primal solution (speciation), activity coefficients, amounts and bulk compositions of phases.



2. Input Files Needed to Run a Standalone GEMIPM2K Calculation


The test examples of usage of GEMIPM2K on the TNode and TNodeArray levels can be executed with the following command line:

gemnode.exe <TestDir>/<TaskName>-dat.lst > <TaskName>.out

where <TestDir> is an optional subdirectory path; <TaskName> should be substituted by a string denoting the problem name, e.g. "MyTask". The file <TaskName>.out will contain console output of the program, redirected there during execution of the TNode example. Without this redirection, the output can be found in a terminal (Linux) or lost (Windows).

The simplest example of the command line (under Windows) to call the GEMIPM2K example on TNode level looks like

gemnode.exe MyTestData/mytask-dat.lst > mytask.out

The file <TaskName>-dat.lst must have the following contents (also automatically created by default upon file generation from the GEMS-PSI code):

-t "<TaskName>-dch.dat" "<TaskName>-ipm.dat" "<TaskName>-dbr-<node0>.dat"[, "<TaskName>-dbr-<node1>.dat", ... ]

This is simply the list of all files prepared for the GEMIPM2K input; all these files must be located in the same directory where the *-dat.lst file itself is located. 

The -t command-line key tells that the following <TaskName>-dch.dat, <TaskName>-ipm.dat, <TaskName>-dbr-<node0>.dat files, and so on, are prepared in free-text input format. Alternative command-line key is -b (the files are prepared in binary format - this can only be done from GEMS-PSI code, or the files were written from the GEMIPM2K). In this case, when the file set is written by GEMS code, the file <TaskName>-bin.lst will be generated having the following content:

-b "<TaskName>-dch.bin" "<TaskName>.ipm" "<TaskName>-dbr-<node0>.bin"[, "<TaskName>-dbr-<node1>.bin", ... ]

The first file in the list, <TaskName>-dch.dat, (or <TaskName>-dch.bin) contains input data for the chemical system definition (CSD) that can be used by both GEM and the mass transport parts of the coupled code (see example in the file dch-dat-doc.txt). These data are also needed for reading/writing the *-dbr.dat files (see below).

The second file in the list, <TaskName>-ipm.dat, contains input data that are used internally by the GEM IPM kernel and must not be exchanged with the mass transport part. This file also contains the flags and values for the numerical setup of GEM IPM subroutines, such as the convergence threshold.

There can be one or many comma-separated file names of *-dbr.dat  (node description) type, which will be read sequentially. On the TNodeArray level, these files contain composition and speciation of all chemical systems needed to set up the initial conditions for the mass transport modelling. On the TNode level, in principle, one *-dbr.dat file is sufficient; if there are many such file names then they will be read one by one, but only the last one will remain in the work DATABR structure in the program memory. See an example of *-dbr.dat file in the file dbr-dat-doc-txt.

To summarize, the GEMIPM2K will read input text files specified in the command line in the following order:

<TaskName>-dat.lst
<TaskName>-dch.dat
<TaskName>-ipm.dat
<TaskName>-dbr-<node0>.dat
............
<TaskName>-dbr-<nodeN>.dat

As an option, the files can be read in in the binary format ( -b key in the file <TaskName>-bin.lst) in the following sequence:

<TaskName>-bin.lst
<TaskName>-dch.bin
<TaskName>.ipm
<TaskName>-dbr-<node0>.bin
............
<TaskName>-dbr-<nodeN>.bin

The whole collection of GEMIPM2K input files describing the chemical system for starting the coupled examples can be written directly from the GEMS-PSI code (either in GEM2MT module or, for a single equilibrium system, from the GEMS Single Thermodynamic System dialog, the "Data" "Write IPM ..." menu command).

Later on, the text input files can be modified by the user according to descriptions of free-input file formats found in the accompanying files dch-dat-doc.txt, ipm-dat-doc.txt, and dbr-dat-doc.txt.

The format of these file types is essentially the same, only the data tags and fields differ. The file consists of two sections - the "dimensions" section where the data needed for memory allocation in GEMIPM2K are provided, and the "free-data" section which follows next. The two sections are separated by a keyword tag

<END_DIM>

Many data object entries can be skipped (especially in the *-ipm.dat file), in which case the default values will be used automatically by GEMIPM2K. The data objects without default values must always be provided at input (they are marked accordingly in the file format documentation files).

  

2.1. The DCH File with Chemical System Definition

The DCH text input file should be read before the IPM file and the DBR files.

The file <TaskName>-dch.dat contains parts of chemical system definitions (CSD) that can be requested both by GEM and by the FMT parts of the coupled code. Contents of this file will be placed in two memory data structures: (1) DCH structure visible from both GEMIPM and the FMT part and (2) MULTI structure - the internal work structure of GEM IPM, not visible from the FMT part. The <TaskName>-dch.dat (and the DATACH structure) also contain dimensions and index lists that are needed for reading and accessing the DBR (node description) files like <TaskName>-dbr-<node0>.dat. Finally, the DCH file and structure contains grid arrays of thermodynamic data used later on for automatic interpolation of gibbs energies G0, molar volumes V0, absolute entropy function S0 and other look-up grid arrays of thermodynamic data  of components to given values of temperature T and pressure P in GEM IPM calculations for any node and time.

The DCH text input file consists of nine sections, of which sections (1), (2),  and (3) contain dimensions needed for memory allocation must appear before the <END_DIM> tag. Any other data object fields can appear after this tag in any order, and some data objects can be skipped. If the data object tag is not found in the input file, its elements will be assigned default values, if possible.

It is recommended to place the input data  in logical sections as in DCH files printed from the GEMS code. The section (4) contains a DATABR structure configuration; section (5) contains names and some data for independent components (IC); section (6) contains names and data for the dependent components (DC); section (7) contains names and data for phases; section (8) is the stoichiometry matrix A for all Dependent Components in the system, and section (9) contains the lookup arrays for interpolation of thermodynamic data for DCs.

2.2. The IPM File with GEM IPM Numerical Settings, Optional Sorption and Solution Model Parameters

 The IPM text input file for GEM-IPM specific data is usually generated from GEM-Selektor package. The GEMIPM2K program reads this file after the DCH file and before any DBR file.

The file <TaskName>-ipm.dat contains additional data and numerical controls from the IPM (MULTI) data strucutre that are necessary for the GEM IPM algorithm but are not relevant for the FMT part. As an option, the whole binary MULTI structure can be read from the <TaskName>.ipm binary file if the -b key is set at the beginning in the <TaskName>-dat.lst file.

The IPM file is complementary to the DCH file. The IPM file contains parts of chemical system definitions (CSD) that are  used by GEM IPM part but should not visible to the FMT part of the coupled code.

The IPM text input file consists of eight sections, of which  section (1) contains numerical controls of GEM IPM algorithm operation  and convergence; section (2) contains dimensionalities for additional memory allocation for surface complexes (main dimensionalities of the  chemical system are given in DATACH file); section (3) contains tolerances and controls of the numerical behavior of GEM IPM-2 kernel; section (4) defines the presence and the data for mixing models of multi-component phases and,  in particular, the arrays of non-ideality parameters; sections (5) and (6) provide some complete data arrays that may be incomplete in DATABR file(s); section (7) contains some initial data for phases; and, finally, section (8) contains initial data for sorption (surface  complexation) phases, present if set so in section (2).

Sections (1) and (2) contain flags and dimensions needed for optional memory allocation which must appear before the <END_DIM> tag. Any other data object fields can appear after this tag in any order, and some data objects can be skipped. If the data object tag is not found in the input file, its elements will be assigned default values, if possible. It is recommended to place the input data  in logical sections as printed from GEMS code.

2.3. The DBR File with Node Data Bridge Composition and Speciation Data

The DBR text input file for DATABR (node recipe) data can also be generated by GEM-Selektor. The GEMIPM2K program reads such files after the DCH and IPM files. The DATABR data structure comprises the interface ("data bridge recipe") for coupling mass transport and chemistry (see DATABR.H file).

The file <TaskName>-dbr-<node0>.dat (read into the DATABR memory structure) contains parts of the chemical system definition that may be changed by the FMT part (input data of GEM such as T, P, bulk composition, metastability constraints) or are returned after the GEM IPM calculation and be utilized by the FMT part on the next time iteration. The <TaskName>-dat.lst file may contain any number of (comma-separated) DBR file names. These files will be read one-by-one: (a) into the same work DATABR structure on the level of TNode class for coupling of GEMIPM with already existing FMT codes; (b) distributed into different DATABR structures for many nodes on the level of TNodeArray class  (and coupled FMT-GEM codes written on the basis of TNodeArray); see documentation to the TNodeArray example for more information in this respect. 

As an option, the DCH and DBR files can be read in in the binary format (after changing the second -t key to -b in the file <TaskName>-dat.lst).

The DBR text input file consists of several logical sections. Dimesions of all arrays in DATABR correspond to those given in Section 2 of DCH file, and the order of components and phases - to the index lists in Section (4) in the DCH file. Here, we will refer to tags from these two sections even though these tags are not present in the DBR file; it is assumed  that the DATACH structure exists and is correctly filled out before any operation with DBR files or DATABR structures.

The DBR text input file consists of six  sections, of which  section (1) contains the controls of GEM IPM operation and data exchange; section (2) contains the chemical system scalar variables; section (3) contains the FMT scalar variables; section (4) contains the data for Independent Components; section (5) contains the data for Dependent Components; and, finally, section (6) contains some data for phases.

There is no <END_DIM> tag in the DBR file. After section (1), any data objects can appear in any order, and some data objects can be skipped. If the data object tag is not found in the input file, its elements will be assigned default values, if possible. It is recommended to place the input data  in logical sections as in DBR files printed from GEMS code.


3. Free Data Input File Format of GEMIPM2K

This section summarizes the syntax of free data input format used in text DCH, IPM and DBR files readable by the TNode class subroutines.  Details about data objects for each file are given in the respective sections above, as well as in the text-level documentation example files dch-dat-doc.txt, ipm-dat-doc.txt and dbr-dat-doc.txt.

The file consists of one or more input data objects (fields) and optional comment lines. A comment line begins with the '#' character in the first position; by reading the file, anything after '#' will be ignored until the line end code(s).

Any data object field begins with a tag - an identifier in angle brackets <>. The tag is followed by one or more values, depending on the type and dimensions of the respective data object. The values are space-, tab-, or endline -separated; the numerical values can be given as (signed) integer, floating-point, or decimal exponent (with e or E) forms; the string values must be taken in quotes ' ' or " ", e.g. 'Quartz'.
An example of the data object with comment lines is given below:

# Bulk composition of (reactive part of) the system - GEM input (moles)
# Number of elements: <nICb>; order of IC: defined in <xIC> index list
# in the DCH-DAT file.

# This vector is the main input of GEM IPM. Inconsistent data in <bIC>
# may cause bad convergence or failure of the GEM IPM algorithm.
# *** This data object cannot be skipped from the DBR-DAT file!
<bIC>
3.31e-04 3.31e-04 2.0e-07 110.6837 1.0e-07 55.34286 0
# Numbers in <bIC> must be positive, between 1e-16 and 9.9999e+5 mol,
# except the last number (total charge) which is usually set to zero.


The data objects which define dimensionalities of other data objects must always be provided in the beginning of the file, before the reserved tag <END_DIM>, or in another file which had been read before this file. In relation to the above example, the <nICb> item has been provided in the companion DCH file, where it looks like:

# *** Number of Independent Components kept in the DATABR structure, nICb <= nIC
<nICb> 7


To see which value in the <bIC> vector corresponds to which Independent Component, check the <xic> index list and the <ICNL> name list in the DCH file:

# DATACH index list of Independent Component kept in the DATABR structure.
# *** xIC Size: see <nICb>
<xIC>
0 1 2 3 4 5 6

.......

# Names in the ICNL list must be alphabetically sorted. Size of the list: see <nIC>.
# *** Data in all other arrays involving ICs must follow the order given in this list.
<ICNL>
'C' 'Ca' 'Cl' 'H' 'Mg' 'O' 'Zz'

So, in this example, the <bIC> data object in the DBR file can be represented like this (the second line is a comment for readability)::

<bIC>  3.31e-04  3.31e-04  2.0e-07  110.6837  1.0e-07  55.34286  0
#        'C'       'Ca'     'Cl'      'H'      'Mg'     'O'     'Zz'


 After the <END_DIM> tag (present in DCH and IPM files but not in DBR files), other data object items may follow in any order, although it is recommended to keep them in logical sections as arranged in the files printed from the GEMS code.

Some data objects can be skipped (the tag together with all values; part of values cannot be skipped). Many data object fields after the <END_DIM> tag in IPM and in the DBR file are not mandatory; when skipped, the default values will be set in the program memory. However, if some optional dimensions are set (e.g. <Lads> before the <END_DIM> tag in the IPM file) then the data objects connected to it must also be provided in the rest of the file.

More comments for <NodeStatusCH>: If GEMIPM calculation is called when in the DATABR structure this code is set to 1 then the simplex automatic initial approximation will be used. This results in a larger number of GEM IPM iterations (typically between 20 and 100) but guarantees that the correct stable phase assemblage will be found. Alternatively, setting the code to 5 causes GEM IPM to use the available GEM solution (vectors 'xDC' and 'gam' in this file) as the initial approximation. This may dramatically reduce the number of iterations (typically to 1 - 5) but does not guarantee that the possible appearance of a new phase (e.g. upon change of T or bulk composition) will be detected. The normal return codes for both modes are, respectively, 2 and 6. It is advisable for the FMT program part to check these codes in this field before continuing because any other code value except 1,2 and 5,6 signals an error in GEM calculation (see NODECODECH in DATABR.H for details).
