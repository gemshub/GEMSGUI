1. Introduction - Why Scripts?

Interpretation of scripts or macros in large program packages (text processors, web browsers, or CAD systems) is an efficient tool to reach a compromise between the hard-coded functionality of the program and the unforeseeable variety of the user's needs and ideas. This applies in full to geochemical modelling program packages such as GEM-Selektor. The main GEM IPM algorithm is hard-coded for the best performance, as well as the most common techniques for temperature/pressure corrections of thermodynamic data and ionic strength corrections for aqueous species. Any addition or change in the hard-coded methods requires re-compilation and re-linking of the code; this is not always an easy job for developers, but far more difficult and time consuming for the majority of end-users.

For instance, there is a variety of possible mixing models of mixing for the multi-component phases, many of which are available (hard-coded) in TSolMod class library (more...), but it is not possible to hard-code all models, also those just appearing in the literature. For these reasons, the user must have a possibility to encode in a simple way, right in the Phase database records, the most flexible calculations using any data that will exist at the run time in the program memory. In this way, the users can extend the database and "teach" the program package without any need to re-compile it.

In general, there is also no way to hard-code in advance all possible meaningful "process simulation" scenarios that may be of interest to (geo)chemists. Hence, in the GEM-Selektor code, simulation of any irreversible process (as a series of local/partial equilibrium states) can be defined as a process script (part of the Process record) where the user writes what exactly and where should be changed in the system definition (stored as SysEq records) to produce the next step of the simulated sequence of equilibrium states.

To  watch what has been actually calculated, the user must also be able to tell the program what output data to collect,  display (ideally, in comparison with some experimental points), copy-paste, and print. Scripts can be very helpful in accomplishing such needs; the "data sampling" scripts (in RTparm, Process and GtDemo records) provide maximum flexibility and performance in collecting only relevant results, thus saving a lot of time - especially if many modeled points were generated for a large chemical system.

Scripts can also be embedded into the print/export and import format scripts of GEM-Selektor. This should provide the user with even more freedom and flexibility because all numeric input and output data objects visible on the module window pages are accessible to format- and math scripts via their labels and indices. The user can organize the output of relevant data only, scaling them to the chosen units of measurement, and presented in the required format.

In the forthcoming versions of GEM-Selektor code, the syntax of math scripts will be updated for the conformance with the ECMAscript standard (already used, for instance, in JavaScript). In this case, a conversion utility for the present-style scripts will be provided.

 

2. GEM-Selektor Built-In Script Interpreter

GEM-Selektor contains a built-in interpreter that converts the scripts (i.e. texts containing user-defined operators and expressions) into an internal binary code. This code is then automatically executed when necessary during the program operation. 

The GEM-Selektor scripts are written in a rather simple language that combines features of Basic and C programming languages. The only kind of identifiers that can be used in scripts are labels of data objects, usually visible on GEM-Selektor module window pages. Through these labels with optional indexes, the math scripts can access numerical data (integer, float and double precision formats are automatically detected) arranged as scalars, line- or column- vectors or matrices. Very limited set of operations can be performed also on ASCII string objects.  No identifiers other than valid data object labels (up to 6 characters long) are allowed in math scripts; in other words, the math script interpreter cannot create new names of variables, although it can create any number of numerical or logical constants. Many built-in mathematical functions are available within script expressions.

There are three stages of processing a source script.

  • At the first (preprocessing) stage, the script text will be checked for the presence of explicit names of Independent Components (IComp), Dependent Components (DComp or ReacDC), Phases, and Predefined Composition Objects (Compos)  in place of indexes. If such a name is found, the respective name list will be searched through until the numerical index for this component, phase, or PCO is detected. Then the name will be replaced by the numerical index (which is often unknown before the actual calculation begins).

  • At the second (translation) stage, all math script operators will be parsed, analyzed, converted into the inverse Polish notation code and table of constants, and stored in an internal binary data structure. In the case of a syntax error, a dialog box will appear, displaying the remaining unprocessed part of the source script. In most cases, translation will be resumed after the user has corrected the error(s).

  • At the third (execution) stage, the run-time calculation will be performed using a stack-based algorithm and the binary (translated) code, whenever necessary. Run-time errors (such as zero-divide), in principle, may produce invalid results or crash of the code; fortunately, this rarely happens because in most cases, the error exceptions are generated, and error messages will be displayed.

 

2.1. Data Object Types and Labels in GEM-Selektor

The GEM-Selektor GUI (Graphical User's Interface) can access any data object that exists in the program memory and has a "Data Object Descriptor" (DOD). DOD is a small structure containing a label (up to 6 ASCII Latin1 characters) of the data object, a pointer to its memory location, information about the type, cell size, number of cells in the column and in the row, and a link to a short comment (tooltip). Creation of DODs is hard-coded, but their contents can be changed in some cases at runtime, during the program operation (for instance, at remake or input of database records).

There are two kinds of data objects (and, accordingly, DODs): static and dynamic. Static data have the same location, type and size during the whole program execution time, while dynamic data can be re-sized, moved in memory, and even converted into another data cell type. The latter can be numerical (integer, float, double), fixed- and variable-size strings, and one-byte type. This information can be obtained from the Help window index.

In GEM-Selektor, the DOD array for more than 1000 data objects makes a bridge between the GUI window pages, the internal data structures, the database records, the runtime HTML help system, and the interpreters of math scripts, print/export, and import scripts. The DOD labels with indices serve as identifiers in script expressions and other kinds of scripts. Skipped index is interpreted as zero index; hence, a DOD label written without indexes will be interpreted as the same label with two zero indexes. Each index is counted from 0 to N-1 or 0 to M-1, where N is the number of rows and M is the number of columns in the (numerical) data object. Both N and M are known in any valid DOD (i.e. any DOD that points to a valid memory location).  

 

2.2. Where Scripts Can be Used

Scripts can be used in definitions of the non-ideal multi-component phases (Phase records, DcEq and PhEq  text editor fields), process simulators (Process records, P_expr and  pgExpr text editor fields),  graphic/table demo samplers (GtDemo records, gExpr text editor field), and thermodynamic data tabulators (RTparm records, tExpr text editor field). These record types embrace the most versatile data in thermodynamic modeling. In addition, math scripts can be embedded within the print-, export- and import format scripts. 

Like any programming, writing math scripts requires some experience, understanding of principles of operation, and some care to avoid unnecessary debugging or program crash. The best way to learn efficient programming is to read and modify working examples. For this reason, GEM-Selektor program package is distributed with a library of template math scripts. This library (in form of SDref records) contains a collection of some typical solutions and is very easy to extend: every user can open a script record, save it under a different record key, and modify as necessary. The users are encouraged to share their favorite template math scripts with other users and even with the developers.

To facilitate writing scripts even more, several create/remake wizards can automatically assemble simple scripts upon selection of items in selection lists. For instance, this can be done in RTParm Wizard, Process Wizard, and GtDemo Wizard. Thus assembled scripts can be immediately executed after the Wizard is finished by clicking "Calculate" button. Any script can be edited/enhanced by editing it in the respective module window page. If such edited scripts exist in the database record, the respective wizard pages should be skipped upon remake, otherwise the edited contents will be lost. 
 

3. Script Syntax and Operators

  Any script is a text containing optional comment lines and a sequence of one or more operators of a general form:

< variable > =: < arithmetic expression > ;

here, =: stands for an assignment operator; semicolon  ;  marks the end of expression.

Result of  < arithmetic expression > is a double-precision numerical value. After calculation, this value is assigned to < variable > (i.e. written into memory location pointed by  < variable >); data type conversion  is performed automatically, if necessary. 

Any text beginning with the  $  character until the end of line is considered as a comment and will not be translated. Comments may occur anywhere in place of space characters. Any number of comment lines is allowed.

All variable identifiers (DOD labels), keywords, and function names are case sensitive.

 

3.1. Arithmetic Expressions

To specify an arithmetic expression, the variables (identified using DOD labels only), constants,  built-in functions and operations can be used.

Brackets ( ) serve for changing the default priorities of operations.

Syntax of an arithmetic expression (in BNF recursive notation, \ \  contain alternatives separated by | symbol "or", braces {} contain optional items that may or may not exist ):

< arithmetic expression > ::= < term > { \ + | - \ < term > }

< term > ::= < pfactor > { \ * | / \ < pfactor > }

< pfactor > ::= < factor > { ^ < factor > }

< factor > ::= ( < arithmetic expression > ) |
               < constant > | < variable > | < function > |
               < logical expression >

Arithmetic operations:

+ addition
- subtraction
* multiplication
/ division
^ power


All numerical operands are converted into double-precision floating-point numbers before applying arithmetic operations. The result is also a double-precision number.


3.2. Variables

Any variable must begin with an  identifier  (i.e. name up to 6 alphanumeric characters, beginning with a Latin letter), which is interpreted as a DOD label existing in the GEM-Selektor program at the moment of translation and execution of the math script. Any other names will be considered as errors (i.e. math scripts are not allowed to create new variables). There are  three forms of variables that may be used in math scripts:

  • scalars (example: pH  )

  • elements of vectors ( example:  x[j] )

  • elements of matrices (example:  a[j][i] )

Data types are automatically determined from the data object descriptors (DODs). Formal syntax in BNF notation is:

< variable > ::= < identifier > | < identifier > [ < index > ] |  
                 < identifier > [ < index > ] [ < index > ]

< index > ::= < arithmetic expression >

Identifier  is a sequence of up to 6 Latin letters (including underscore symbol) and digits beginning with a letter. Math scripts are case sensitive, so  pH  and  Ph  are different identifiers. 

Index    is any arithmetic expression; only integer part of it (rounded to the closest integer) will be used. 

Variable   cannot have more than two indices.

If only one index is written then it is considered as an absolute (one-dimensional) index even for matrices.

Matrix elements are counted row-by-row, as in C language.

Absence of an  index  is regarded as zero absolute index, that is, the variable refers to the first cell of the data object. 


3.3. Functions

The script interpreter permits usage of built-in functions in all expressions. As arguments, either arithmetic expressions or intervals can be specified.

Interval  is a construction of the form  x[ib:ie], which tells the function to process all cells of a vector or matrix with indices fitting inside of   ib:ie  interval, including  ib  and  ie  indices. Syntax in BNF notation:

< function > ::= < function identifier > ( < argument > ) |
         < function identifier > ( < argument > , < argument > )
< argument > ::= < arithmetic expression > | < interval > 

< interval > ::= < identifier > |
                 < identifier > [ < index > : < index > ] |
         < identifier > [ < index > : < index > ][ < index > : < index > ]

Number of square brackets with indices or intervals cannot exceed two. By default, [ 0:N-1 ] or [ 0:M-1 ] intervals are taken, where  N  or  M  are current dimensions in the data object descriptor referenced by the identifier. 


3.4. Logical Expressions

Internal predefined constants  ONE_  (true) and  NUL_  (false) are used for logical values in math scripts. Logical value NUL_ is equal to the numerical value 0.0. Any other value (greater than 1e-34) is considered as ONE_. 

When a logical expression is calculated, any result different from   NUL_   is considered as ONE_ (true). The syntax is:

< logical expression > ::= ( < condition > ?
               < arithmetic expression > : < arithmetic expression > )

< condition > ::= < term > { \ & | | \ < term > }

< term > ::= < relation > | ( < condition > ) | !< condition > |
               < arithmetic expression >

< relation > ::= < arithmetic expression >
               \ < | <= | > | >= | = | <> \ < arithmetic expression >

Logical operations (in order of decreasing priority):  

!
 not  
&
 and    
 |
 or    


Comparison operations:   

<   less than               <=  less than or equal        
> greater than >= greater than or equal
= equal <> not equal


Attention! Logical expression must always appear in brackets () in math scripts.


3.5. Conditional Operator, Block, and Loop Operator

In addition to usual operators:

< operator > =:: < variable > =: < arithmetic expression > ; |
                 < variable > =: (< logical expression >);

scripts can contain conditional operators of the form:

< opcond > =::   if( < condition > ) 
                    < block >
                 else < block >

Here,  the line beginning with  "else"  keyword is optional and can be dropped. 

< block > =:: begin < operator > { < operator > } end |
              begin < opcond > { < operator > } end

< block >     s a sequence of one or more operators between the  begin  and  end  keywords.
< opcond >  may be part of the block, i.e. conditional operators can be nested (up to 3 levels in the present implementation).

When a construct

if ( < condition > )
   < block ONE_ >
else < block NUL_ >

is processed, first, the value of  <condition>  is calculated. < block ONE_ >  is executed only when  < condition >  is not  NUL_. Otherwise, < block NUL_ >  will be executed, if present.

Loop operator  is available in the simplest form:

do ( < condition > )
    < block >

Upon execution, a value of < condition >  after the  "do"  keyword is calculated first.  If it is not  _NUL  then  < block >  is executed,  < condition >  is checked again, and so on until < condition >  will be equal to  _NUL. After that, the operator following the end of  < block >  will be executed.

 

3.6. Priority of Operations

Operations are listed from the highest to the lowest priority. In one line, operations have the same priority.

!     logical  "not"   (unary)
&     logical  "and"  (binary)
|     logical   "or"   (binary)

=  equal to;   <>  not equal to;  <  less than;   <=  not greater than;   >  greater than;  >=  not less than

-    inversion of sign  (unary)
^    raise to power    (binary), equivalent to pow(x,y) function

*    multiply;   /  divide    
+    add;      -  subtract

Sign inversion must be always written in brackets () together with its single operand.
Example: correct  (-x[j]),  wrong  -x[j].

 

3.7. Constants

Only integer or real numerical constants without sign are allowed in arithmetic expressions.

Negative numbers should be expressed as 0-<constant> or using the inversion of sign operation, e.g. (-25).

Examples of numerical constants:

235 1.928 .000545 2e4 1.84478e-11 (-18) (-2.214e-6)

A special numerical value EMPTY_ (no data) can be assigned using a function  empty(). 
This function returns an internal EMPTY_ constant and thus can be used in assignment and comparison operators.

Internal logical constants:  NUL_  (false)   ONE_  (true).  NUL_ can be entered directly as 0;  ONE_ as 1.

<Under construction>:  symbolic constants.
Symbolic constants are strings of the form

"<s1>...<sn>"

or sequences of ASCII characters <s1>, <s2>, ... <sn>  in quotation marks "".
Such constants can be used only in comparison operators  = or <>, and in the assignment operator  =: .

In this version, only the first character of a string is used as an integer number.
In other words, the actual comparison occurs as  <s1> = <s2>  or  <s1> <> <s2>. 


4. Script Built-In Functions

Various built-in functions  can be used in GEM-Selektor scripts. These functions are hard-coded, some of them can also be called from within the source code. There are three classes of functions:  those with arithmetic arguments; with interval arguments; and with index or without arguments (mainly chemically-specific functions). 

During execution of scripts, such common functions as logarithms ( ln, lg ) or square root  ( sqrt ) are called directly from C/C++ floating point math libraries and may return errors or exceptions if the argument value is wrong. Even though these exceptions are caught in the code of script calculator, they still can occasionally cause abnormal behavior or crash of the program.

Before calling a function, its argument(s) will be evaluated yielding double precision floating-point value(s). All functions  return a double precision result. In case of error, all functions return 0. In the list below, x stands for any arithmetic expression allowed in math scripts

sign( x ) Sign of x (returns +1 or -1)
exp( x ) Exponent of x; e ^ x ( -85.196 < x < 85.196 )
sqrt( x ) Square root of x ( x > 1e-37 )
ln( x ) Natural logarithm of x ( x > 1e-37 )
lg( x ) Decimal logarithm of x ( x > 1e-37 ). Alternative:   log( x )                          
sin( x ) Sine of x ( x in radians; x < 2^52 )
sinh( x ) Hyperbolic sine of x
cos( x ) Cosine of x ( x in radians; x< 2^52 )
cosh( x ) Hyperbolic cosine of x
tan( x ) Tangent of x
tanh( x ) Hyperbolic tangent of x
asin( x ) Arc-sine of x (-1 <= x <= 1 )
acos( x ) Arc-cosine of x (-1 <= x <= 1 )
atan( x ) Arc-tangent of x
ceil( x ) Closest integer value greater than x

5. Indexation in GEM-Selektor Lists and Script Preprocessing

Name-Index Substitutions

Data structures for calculation of equilibria in GEM-Selektor are very flexible; any Independent Component, Dependent Component (species), or Phase can be excluded from the run-time GEM IPM problem by turning it off in the system definition. For this reason, a numerical index of a particular species or phase may vary from one IPM task to another. Hence, it would be  very inconvenient to use explicit (numerical) indexes in math script equations in the Phase, Process and GtDemo records. Instead, a big advantage would be to use names of species or components directly,  instead of numerical indices, wherever possible. 

To make scripts easier to read and compatible between different system definitions, a pre-processing step is performed before the actual translation and execution of the script. During pre-processing, the names of phases, species and compositions are identified, extracted and converted into into their actual run-time indexes. This is done automatically because the program knows which list of names is to be searched for a given data object label (coded in a configuration file  vis_od.ini). All the user has to do when writing the script is to put the species- or phase name in braces in place of the numerical index value, for instance:  x[{CaCO3@}]  instead of  x[24] .

During calculation of equilibria, two indexation systems exist in the data structures of GEM-Selektor. One system is defined by the full lists of Independent Components, Dependent Components, Phases, and PCO for the currently open System definition in the Project data structure. This indexation is used also in the vectors for thermodynamic data and system configuration.

Usually, some elements, species and/or phases are switched off and must be eliminated from the run-time lists.  Therefore, another set of (usually smaller) lists of Independent Components, Dependent Components, and Phases is automatically created within the GEM IPM run-time data structure according to the configuration switches in the system definition (i.e. in currently loaded SysEq record).  The second indexation system is also applied when a SysEq record is expanded for the GtDemo data sampling. 

Both the full (Project) and the condensed (IPM) indexation systems are connected by three special data objects  mui, muj and mup that belong to the IPM runtime data structure. Given that i, j and k are indexes of an Independent Component, a Dependent Component and a Phase, respectively, in the IPM runtime structure, the corresponding indexes I, J, and K for these components and phase in the System definition can be calculated as follows: I = mui[i]; J = muj[j]; K = mup[k].

For example, let us assign a value of normalized chemical potential of Ca+2  ion to a Qd[2] work cell, assuming that Ca+2 is the sixth species of the aqueous phase in a particular equilibrium problem. This can be written in the script as the following operator (remember that the indexation begins from 0):

Qd[2] =: fMu[5];

But this writing is valid only for this particular case. If some species will be turned off or on, or the script will  be used in another modeling project, the index of Ca+2 ion may become different, and the above operator will produce a wrong result. A more safe and general expression (to be preprocessed) looks like this:

Qd[2] =: fMu[{Ca+2}];

The preprocessor will locate the string in braces ("Ca+2") and look for the first match of it in the run-time list of species names DCnam (in this example, the match will occur in 6-th line of the list). Then the sub-string {Ca+2} will be substituted by 5, yielding Qd[2] =: fMu[5]; and later on, the operator will be translated as usual.

Another example: suppose that gypsum (CaSO4(H2O)2) is included as a Dependent Component both in the System definition and in the IPM runtime structure. Then the stoichiometry coefficient of Ca in gypsum can be obtained as:

Qd[3] =: A[Ca}][{gypsum}];

Standard molar Gibbs energy of gypsum, taken from the thermodynamic data vector in the modelling project structure, will be:

Qd[4] =:  gTP[{gypsum}];

or, alternatively (nested index expressions are allowed in math scripts):

Qd[4] =:  gTP[ muj[{gypsum}] ];

The normalized standard molar Gibbs energy of gypsum, kept in the thermodynamic data vector in the IPM runtime structure (Multi), can be extracted in this way:

Qd[5] =: g0[{gypsum}];

Note that the index for gypsum in the IPM runtime structure will most probably be different (smaller) than that for gypsum in the project system definition, if some independent or dependent components were turned off.

Separation of Groups of Expressions

A special pre-processing may be performed for math scripts in the DcEq field of Phase window. This is controlled by a switch  p_solt[0,4] in Phase records.  If this switch is set in a "private" mode then a  separate group of math script expressions must be written for every phase end-member. Each group is closed by "\@" characters that must directly follow the ";" symbol of the last operator in the group. The groups of expressions must be placed in the same order as the species in the pDC_v_ list on Page 1 of the Phase window.

The pre-processing is necessary because, if a j-th species is switched off in the system definition, the respective "private" group of math script operators must be skipped from the math script execution. This is achieved by looking for the next (j-th) "\@" in the original math script text and analyzing whether the j-th end-member is included or not in the IPM run-time data structure. If the end-member is included, the whole group of operators will be copied into the math script text to be translated; otherwise, the program proceeds to the next end-member (and group).


6. Template Script Library in Default Database

GEM-Selektor code comes with a collection of useful examples of math scripts called "template math script library".  

The purpose of this library (that may be extended in future versions, also with users contribution) is to facilitate writing math scripts in your modelling projects. Instead of writing "from scratch", it is much easier to learn by doing, i.e. to select the closest template script, copy it into clipboard (Ctrl-V), paste into the appropriate text field (Ctrl-C), and modify (usually change names of components and phases, or some numerical constants). This is why the library scripts are called "template": in almost all cases, they need modification to adjust the expressions to your particular system formulation. 

The template scripts are located in SDref  records under keys beginning with "mscript" and ending with "phase", "rtparm", "proces" or "gtdemo" names of modules where the scripts can be executed. After opening any such record, you will find the scripts on Page 2 of SDref window. There, you can edit or add scripts or comments and save the record under a different key into the SDref  file of your modelling project. In this way, you can make your own library of your favourite template scripts and, later on, even share it with a colleague or with the GEMS2-PSI Development Team.


7. Error Messages

There are three classes of error messages or warnings that can be generated when GEM-Selektor processes the math script: pre-processing messages; translation messages; and execution messages. In the first two cases, the program will show you the message text together with the remaining (not yet analyzed) text of the math script.

Usually, the user will have an opportunity to make necessary corrections by editing the text and saving the database record with edited script, then the program will try processing it again. At the execution stage, when an error message appears, the calculation will be stopped; you will have to go back to the record with the original script, correct it (or change some data in the record) and try the calculation again.


Preprocessing Messages

E00MSPrep: Identifier (variable) <identifier> is not known.
The <identifier> could not be found in the DOD label list of GEM-Selektor. Please, check writing (DOD labels are case sensitive and cannot be longer than 6 characters!).

E01MSPrep: Unknown indexation base for data object  <identifier>
The <identifier> was found in the DOD label list, but is indicated there as non-indexable (maybe, a scalar or a variable-size text object). Please, check writing of <identifier> in the source math script.  

E02MSPrep: Cannot substitute index for  <name>
Generic error message, caused by errors E04, E14, E05, E15, E06, E16 and E07. 

E03MSPrep: Too long name for index substitution  <name>  (>38 characters)
The name in braces {} is too long. Please, check the source script - maybe, a closing brace } is missing.

E04MSPrep: IC name cannot be found in GEM IPM work data lists... <name>
The currently processed IC symbol in braces {} cannot be found in the GEM IPM runtime structure.  Make sure that this Independent Component is turned on in the System definition, or the writing of IC symbol is correct (case-sensitive!).
 
E14MSPrep: IC name cannot be found in Project system definition lists... <name>
The currently processed IC symbol in braces {}cannot be found in the System definition (Project) list. Make sure that the writing of IC symbol is correct (case-sensitive!). Alternatively, check that the correct data vector label is written in the script (i.e. you do not try to find an IC index for a vector with DC or Phase properties).

E05MSPrep: DC name cannot be found in GEM IPM work data lists...<name>
The currently processed DC name in braces {} cannot be found in the GEM IPM runtime structure.  Make sure that this Dependent Component is turned on in the System definition, or the writing of DC name is correct (case-sensitive!).

E15MSPrep: DC name cannot be found in Project system definition lists...<name>

Translation Messages

E00MSTran: Math Script operator syntax error. 
The preceding math script operator could not be parsed. Check that the syntax is correct. A frequent reason is that a comment line does not begin with a "$" character.

E01MSTran:  Missing \"  in string constant.
String constant is too long, probably a closing " character is missing. Correct the script and try again. 

E02MSTran:  Function <function name> is not defined.
Wrong function name; check it in the functions list (case sensitive), correct the script and try again. 

E03MSTran:  Missing identifier of interval: <text>.
In the interval function operand, the identifier (variable name) is missing. Correct the script and try again.

E04MSTran:  Missing  bracket  ) : <text>.  (also error E34) 
The closing bracket could not be found before the end of operator (";") was encountered. Correct the script and try again.

E14MSTran:  Missing  bracket ( : <text>.  (also error  E24) 
More of the closing brackets was found before the end of expression than the opening brackets. Correct the script and try again.

E05MSTran: Unexpected end of Math Script text. (also errors E15, E25, E35, E45 )
End of source text encountered before the expression could be fully parsed (i.e. until ";" symbol occurred). 
Correct the script and try again. 

E06MSTran: Variable <identifier>  is not known.  (also error E16) 
The previously processed name <identifier> could not be found in the GEM-Selektor DOD name list. Correct the misprint (case sensitive!) and try again.

E07MSTran: Math script contains no operators.
Check that the math script text is provided in the proper field, or remake the record to make sure that no math script processing is requested (for Phase or Process records). 

E08MSTran: Assignment needs =:  <text> 
A frequent misprint - the comparison operator "=" is typed on place of the preceding assignment operator "=:". Correct the script and try again. 

E09MSTran: Expression is needed here:  <text>  (also errors E19, E29)
The ";" symbol occurs immediately after the assignment operator, the expression is missing (probably, commented out). Correct the script and try again.
 
E10MSTran: Identifier (variable) is needed here:  <text>
The variable name is missing before index expression (in []). Correct the script and try again. 

E11MSTran: Invalid characters at the end of expression:  <text>
Please check the script; if the script is Ok, report this error to the developers.

E12MSTran: Missing keyword  begin :  <text>  (also error E22)
Please, reconsider the script text after if(), else or do() operators, put begin or end keywords where necessary, and try again.
 
E13MSTran: Missing keyword  end :  <text>    (also error E23)
Please, reconsider the script text after if(), else or do() operators, put begin or end keywords where necessary, and try again.

E19MSTran: Operation stack overflow. (also error E29) 
The most probable reason is a negative numerical constant without brackets. If not, report this error to the developers.

There is no match of brackets within an expression or an if() operator. Correct the script and try again.
