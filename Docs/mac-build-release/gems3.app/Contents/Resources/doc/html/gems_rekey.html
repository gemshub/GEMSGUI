1. Introduction 

The GEM-Selektor database consists of several chains, each containing records of certain type and structure. Each record is stored under a unique record key that consists of several (alphanumeric or numeric) fields containing the most relevant information about the record contents. The program automatically sorts record key lists and can filter them to select subsets of record keys using templates (i.e. "?" or "*" characters in some record key fields).  One or more database files (containing records of the same type) can be linked to a record chain. More about that is written in GEM-Selektor database manual.

The GUI (graphical user's interface) of GEM-Selektor can display record keys in two formats: condensed and expanded.  In the condensed format, the remaining spaces in every field are eliminated, and record key fields are separated by colons ":", for example:  "a:Na:Na(CO3)-:nps:" for the DComp record. This format is economic but not nice for the record key lists - they look much better if record keys are expanded. In this case, no field separation characters are used, and each field is printed in its full length (with spaces added on the right side, if necessary). The above example expanded looks like this: "a   Na      Na(CO3)-        nps ". The program performs automatic conversions between expanded and condensed formats, wherever necessary. 

The diagrams presented below for the record keys of different types consist of two rows. In the first row, the field names with links to explanations are given. In the second row, the type (string str or numeric num) and length (in expanded format) are provided for each field, for instance  str[16]  or  num[5].

As a rule, a record field must contain a string made of Latin letters (case-sensitive!), digits, "-", "+", "@" and "_" characters, but no spaces inside.

No colons ":" can be entered into separate record key fields in "Create", "Clone" or "Save As" dialogs.

The "*" and "?" characters can only be entered within the "Key Filter" dialogs.
 
 

2. SDref - Script- and Data Reference Record Key

 
SD_author SD_year SD_type
str[20] str[5] str[7]

SD_author  field in the case of a bibliography reference contains author's last name, e.g. "Stumm". If there are two authors, write both names separated by a & character, e.g. "Asterix&Obelix". If there are more than two authors or two names occupy more than 19 characters then type the name of the first author followed by "_ea", for instance "Shock_ea". 

If this SDref record contains a script then the SD_author field of its record key must begin with:

  • in the case of a print script: "pscript-[string]";   
  • in the case of an export script: "escript-[string]";  
  • in the case or an import script:  "iscript-[string]"; 
  • in the case of a template math script:  "mscript-[string]".
There, [string] may optionally contain up to 11 non-space characters and can be used as a comment extension, for instance, "mscript-Margules". When looking for scripts, the program checks only the first 6 characters of the SD_author field.

SD_year  field in case of a bibliography reference should normally contain year of publication or completion, for example"1997". If this SDref record contains a script then the SD_year field is actually interpreted as a 4-digit script variant number, for instance "0001". We suggest numbers like "1002" be used for storing the user-created or modified scripts in order not to mix them with the default script library provided with the program. 

SD_type  field usually contains a word (up to 6 characters in lowercase) denoting type of the bibliographic reference or the record type (for scripts). We suggest to keep up with codes such as  "pap" (reviewed paper), "book", "man" (manual), "ms" (manuscript), "rep" (report), "dat" (data file), "prog" (program manual), and so on. If this SDref record contains a script then the SD_type field is checked by the program to select all scripts for a given module (record) type encoded as: "sdref", "icomp", "dcomp", "reacdc", "phase", "compos", "syseq", "proces", "gtdemo". In addition, there are submodules "mtparm", "system" and "multi" needed for special printing scripts related to GEM IPM calculations.
 
 

3. IComp - Independent Component (IC) Record Key

 

IC_symbol

IC_type

IC_comment

str[6]

str[4]

str[22]

IC_symbol  field  must contain a unique identifier (symbol) of  the stoichiometry unit (IComp), which must begin with an upper-case Latin letter, followed by up to five lower-case letters or _ (underscore) characters. IC symbols may appear in chemical formulae of species or compositions.  Examples:

  • Si         silicon
  • H           hydrogen
  • Cit      citrate ligand
  • Alo      Al2O3 oxide unit
  • Vol      volume constraint
  • Zz        (formula) charge


IC_type  field contains isotopic mass or a code internally used in GEM-Selektor calculations or in parsing chemical formulae. Some examples of possible codes:

  • e          chemical element (natural mixture of isotopes, e.g. C:e:Carbon:)
  • f         formula unit (e.g., SiO2,   Sio:f:Si_dioxide:)
  • a          auxiliary IC of unknown stoichiometry (humate, Hum:a:Humate_ligand:)
  • z          electrical charge (Zz:z:Charge:)
  • v          volume (for volume constraint of the system, Vol:v:Volume:)
  • [int]     isotopic mass (e.g.,  C:13:Carbon_13:)

An integer number in the IC_type field denotes an isotope (and is interpreted as the isotopic mass). Thermodynamically, isotopes define separate mass balance constraints and hence are treated as separate IC. In chemical formulae, isotopic masses must be written explicitly, e.g.  Ca/12/CO3. 

Usage of cations, ligands, or petrogenic oxides like SiO2  as independent components is theoretically possible, although not welcome, as it would require a complete re-definition of the chemical thermodynamic database.

IC_comment  field  is not analyzed by the program and serves for better readability of IComp record key lists. Spaces are not allowed inside of the comment (only underscores can be used). See examples above.

 

4. DComp - Dependent Component (DC, Species) Record Key

 
Ph_state  Group_id       Species_name       TDS_code  
str[4] str[8] str[16] str[4]

Ph_state  field  must contain one of the phase state codes that define the aggregate phase state of this Dependent Component (species).
 
 Group_id  field identifies a group of Dependent Components, for instance, aqueous species of a given metal cation. This field of DC record key is designed to facilitate keeping related species together in the alphabetically sorted lists. Any sequence of up to 8 capital letters, digits, "-", "+", "@" and "_" characters is allowed (but no spaces). This field is not analyzed by the program, except alphabetical sorting of record keys.

 Species_name  identifies a Dependent Component entity. This name must be a string up to 16 characters long (letters, digits, "-", "+", "@" and "_ "  characters are allowed, but no spaces). The species name must be unique at least within a phase species list. In the kernel (default) GEMS database,  species names  look like chemical formulae for gases, aqueous and surface species, e.g. Ca+2, CaSO4@ or CO2. Names of minerals (as species names) are  to be written beginning with a lower-case letter. Abbreviations like  alm (almandine), arg (aragonite) are allowed but they are less readable than full mineral names.

This record key field is copied into the species lists in the GEM IPM work structure (Multi). The species names are also analyzed during preprocessing of scripts.

TDS_code  (up to four letters, digits, "-", "+", "@" and "_ "  characters ) serves for identification of a consistent subset of thermodynamic data. Some codes are used in the built-in (default) GEMS database. The user must invent his own codes when adding new DComp or ReacDC records in his project databases. 

TDS_code field can be automatically modified by the program when multiple copying of database records is performed, also in creation of new modelling projects. This is necessary in order to avoid possible database search conflicts: the GEMS database cannot contain two records with identical keys. See more about how the user can control modification of the last field of DComp or ReacDC record keys during copying. 

TDS_code is also checked by the GEM-Selektor data override mechanism. In brief, if several DComp records exist in the project database with only different TDS_code fields (i.e. the first three record key fields are identical) then an alphabetically the first record key will be selected for collecting input data and other records will be silently ignored. The same applies to ReacDC records (but not to a mixture of DComp and ReacDC keys) and to Phase records. In this way, the user can retain the original data from the GEMS default database and replace it in calculations of equilibria with her own data.    

   

5. ReacDC - Reaction-Defined DC (Species) Record Key

Ph_state     Group_id    Species_name        TDS_code 
str[4] str[8] str[16] str[4]

Structure of ReacDC record keys is the same as that of DComp record keys (see above). 

The Species_name field refers to the name of a newly-defined species in this ReacDC record. The data override mechanism applies, except that to processing ReacDC or DComp record keys in the REsDC list.


6. RTParm - Thermodynamic Data Tabulator/Plotter Record Key

Ph_state Group_id Species_name TDS_code Source Name_RTP
str[4] str[8] str[16] str[4] str[4] str[12]


Structure of the first four fields of RTparam record keys is identical to that of ReacDC or DComp record keys (see above). These first four fields simply point to a record with input thermodynamic data that must be tabulated or plotted. The data override mechanism does not apply in this case, i.e. the source record with exactly the same TDS_code field will be selected (or error message generated if the record is missing). 

Source  field tells the program where to look for a record with input data pointed to the previous four fields. Two values can be used:

d  amond DComp records
r  amond ReacDC records

Name_RTP  field serves for identification of this RTParm variant (several RTParm records can be created for the same input DComp or ReacDC record). This must be an aplhanumeric string up to 12 characters long (letters, digits, "-", "+", "@" and "_ "  characters are allowed, but no spaces). 

 

7. Phase Definition Record Key

Ph_state   Group_id   Phase_name   Ph_class   Ph_comment   
str[4] str[8] str[16] str[4] str[16]

Ph_state  field must contain the phase aggregate state code (one lowercase Latin letter). Correct setting of this code is very important, as it defines the way how this phase will be treated in GEM IPM calculations of equilibrium states.

Group_id  field identifies a group of related phases, for instance, "FeO" for various iron oxides. This field of Phase record key is designed to facilitate keeping related phases together in the alphabetically sorted lists. Any sequence of up to 8 capital letters, digits, "-", "+", "@" and "_" characters is allowed (but no spaces). This field is not analyzed by the program (except alphabetical sorting of record keys). Two Group_id names are reserved for the automatically generated definitions of aqueous and gas phase in the Project module ("AQELIA" and "GASMXID", respectively).

Phase_name identifies a Phase entity. This name must be a string up to 16 characters long (letters, digits, "-", "+", "@" and "_ "  characters are allowed, but no spaces). The phase name must be unique within the modelling project database. Two phase names are reserved for the automatically assembled definitions of aqueous and gas phases in the Project module ("aq_gen" and "gas_gen", respectively). Names of mineral phases are  to be written beginning with an upper-case letter. 

This record key field is copied into the phase name lists in the GEM IPM work structure (Multi). Phase names are also analyzed during preprocessing of math scripts.

Ph_class field must contain an extended phase class code, to enhance readability of Phase record key lists (not specifically processed by this version of GEM-Selektor).

Ph_comment  field should contain an alphanumeric string commenting this phase definition (to make the Phase record keys better readable and to affect their order in record key lists).

The Ph_comment field can be automatically modified by the program when multiple copying of database records is performed, also in creation of new modelling projects. This is necessary in order to avoid possible database search conflicts: the GEMS database cannot contain two records with identical keys. See more  about how the user can control modification of the last field of Phase record keys during multiple copying. 

The Ph_comment field is also checked by the GEM-Selektor data override mechanism. In brief, if several Phase records exist in the project database with only different Ph_comment fields (i.e. the first four record key fields are identical) then  alphabetically the first record key will be selected into the System definition, and other records will be silently ignored. The same also applies to DComp and ReacDC records (but not to a mixture of DComp and ReacDC keys) referenced within this phase definition. In this way, the user can retain the original data from the GEMS default database and replace it in calculations of equilibria with her own data.    

 

8. Compos - Predefined Composition Object (PCO) Record Key

PCO_name PCO_type PCO_comment
str[16] str[4] str[28]


PCO_name  field  must contain a unique identifier (name) of  the PCO: a string up to 16 characters long (letters, digits, "-", "+", "@" and "_ "  characters are allowed, but no spaces). This record key field is copied into the PCO name list in the GEM IPM work structure (Multi). PCO names are also analyzed during preprocessing of scripts.
 
PCO_type  field enhances readability of the Compos record key lists. This field should contain one of the recommended PCO type codes. The PCO_type field is not analyzed by the program.

PCO_comment  field enhances the PCO_name field; it is not analyzed by the program and serves for better readability of Compos record key lists. Spaces are not allowed inside of the comment (only underscores can be used). The GEM-Selektor data override mechanism does not apply for the Compos record keys.

PCO_comment field can be automatically modified by the program when multiple copying of database records is performed, also in creation of new modelling projects. This is necessary in order to avoid possible database search conflicts: the GEMS database cannot contain two records with identical keys. See more  about how the user can control modification of the last field of Compos record keys during multiple copying. 


9. Project - Modelling Project Definition Record Key 

Pr_name Pr_comment
str[10] str[38]

 
 Pr_name identifies this modelling project. This unique name (alphanumeric, no spaces, up to 10 characters long) must be given by the user when a new project is created. The name is analyzed by the program to check that all the available SysEq and Process records belong to this project because mixing of such records from different projects is not allowed.

Pr_comment  is an alphanumeric string that comments Pr_name. Must also be given by the user when a new project is created. This field is not analyzed by the program.


10. SysEq - Chemical System Definition (CSD) and Equilibrium Record Key

Pr_name PotMin System_name VarCSD Volume Pressure Temperature VarCon
str[10] str[3] str[12] num[5] num[8] num[8] num[8] num[4]

Pr_name  field identifies the modelling project framework and database where this chemical system definition is created. The program automatically sets this field identical to the Pr_name record key field in the currently loaded Project record. 

PotMin  field encodes which of six thermodynamic potentials is to be minimized when calculating equilibrium state in this chemical system using the GEM IPM algorithm. Only total Gibbs energy G(x) ("G" code) can be minimized in this version of GEM-Selektor; minimization of other potentials (A, H, U, Sp, Sv) is reserved for future versions.  

System_name  identifies a model chemical system definition (CSD) with given sets of independent components, dependent components, and phases selected for calculation of equilibrium states. A new System_name (alphanumeric string up to 12 characters long) must be given by the user when creating or cloning a new SysEq record, in which some components or phases will be de-selected or re-selected (i.e. dimensions of lists in the GEM task will change), or a remake of the system structure will be performed. 

Please, note that in most cases of cloning a new CSD from the existing one, only the T,P, bulk composition, metastability constraints, or thermodynamic data corrections will be modified, but no changes will be done in selections of phases and species to be used in the GEM task. In this situation, we recommend to change either of VarCSD, Pressure, Temperature and VarCon fields to save a modified CSD under a new SysEq record key, but leave the System_name field unchanged.

VarCSD  field  must contain a positive integer number of the current CSD (recipe) variant, usually for a modified bulk composition of the system, or specific surface areas of some phases. This field is automatically modified when new SysEq records are generated by the Process Simulator module, where usually 4-digit numbers are assigned. Modification of the VarCSD field is in most cases done while retaining the same System_name field.  

Volume  field usually contains 0 (zero, default), which means an unconstrained volume of the system - default in Gibbs energy minimization where T and P must be given as factors of state, and the total volume of the system will be computed. In specific cases  such as an indefinite equilibrium between aqueous and gas phase with water vapor at Psat(H2O,g), this field can contain a limiting volume of the system (real number in Litres, dm3), for example 1.23404 or 2.447e3.

Pressure  field (default: 1) defines the pressure of interest P (real number in bars, 1 bar = 100000 Pa) for calculation of an isobaric-isothermal equilibrium state by GEM IPM. Examples: 1; 5000; 3.456; 2e4. Zero (0) in Pressure field means Psat(H2O,g), i.e. saturated vapour pressure of pure H2O; the actual pressure will be automatically calculated from the given temperature T. Thermodynamic data for all species included into the modelling project will be automatically corrected from standard pressure Po to pressure of interest P before GEM task calculation, whenever necessary. 

Temperature  field (default: 25) field defines the temperature of interest T (real number in centigrade, T >= 0 oC) for calculation of an isobaric-isothermal equilibrium state by GEM IPM. Examples: 1; 5000; 3.456; 2e4. Thermodynamic data for all species included into the modelling project will be automatically corrected from standard temperature To to temperature of interest T before GEM task calculation, whenever necessary.

VarCon  field (default is 0) must contain an integer number of the current variant of non-trivial metastability constraints and/or additional corrections to thermodynamic data in the gEx_ vector. This field is also automatically modified to '000' when new SysEq records are generated by the Process Simulator module; as the one-digit '0' is default, this helps to see which records were generated by the Process module.


11. Process - Process Simulation Task Record Key 

Pr_name PotMin System_name VarCSD Volume Pressure Temperature VarTD Pe_name PeType
str[10] str[3] str[12] num[5] num[8] num[8] num[8] num[4] str[18]  str[2]

The first eight fields of the Process record key is the same as that of its parent SysEq record key (see above). This parent SysEq record must exist in the database when a new Process record is created; later on, the Process Simulator will use the parent System in cloning and modifying it to generate the whole sequence of local/partial equilibrium states. In the Process module, these eight fields cannot be edited.

Pe_name  field must contain a unique name of this process simulation task (alphanumeric string up to 18 characters, no spaces) because several Process records may refer to the same parent SysEq record. Pe_name should be  entered by the user when a new Process record is cloned or created.  

PeType  field contains a code (symbol) of the process simulation mode. This helps in readability of Process record keys and also in selecting the default configuration when creating a new Process record. Currently, the following modes are available: 

  • "P"    Sequential temperature and/or pressure change at fixed bulk composition
  • "S"    Direct sequential change of bulk composition and/or constraints (default)
  • "G"   Batch inverse titration sequence for incremented pH values etc.
  • "T"   One arbitrary inverse titration calculation as defined in Process control script
  • "R"   Sequential reactor scheme, uses equilibrium bulk compositions of phases
  • "L"   Lippmann diagram (transposed) for a binary solid solution only

Other codes may be added in future versions. If a letters are entered that is not listed above, the program will assume the default "S" mode.

12. GtDemo - Generic Data Tabulator/Plotter Record Key 

Pr_name    DB_chain GD_name     VarGD   GD_comment   
str[10] str[8] str[16] num[4] str[26]

Pr_name  field identifies the modelling project framework and database where this GtDemo sampler is created. The program automatically sets this field identical to the Pr_name record key field in the currently loaded Project record. 

DB_chain  field indicates the top-level record type (database chain) for the data sampling into GtDemo tables.  The user can put in this field either a  database chain name ("proces" or "syseq" or "dcomp" etc.) or a two-letter code ("PR" or "SY" or "DC" or "RE" , as suggested in the tooltip). These codes are not analyzed by the program and only serve to identify what data will be sampled. During creation of a new GtDemo record, the user will be asked to select the top-level record type from the list.

If a record from the top-level chain directly refers to lower-level records, those will be also sampled one-by-one. An important example is Process record which usually contains a list of generated SysEq record keys. In this case, the SysEq record list will be (the whole or in part) taken into GtDemo execution and sampled in a straightforward sequence (using the jR  index in G_expr math script).  

GD_name  field should contain a unique (alphanumeric, no spaces) name of this GtDemo sampler. The name should be entered by the user when creating or cloning a new GtDemo record. Clear, long enough and concise names will facilitate reading of the record key lists and finding the necessary results.

VarGD  field defines a GtDemo variant number (unsigned integer up to 9999) because several GtDemo samplers can be based on the same selection of GEMS project database records.

GD_comment  field may (and should) comment on the two previous fields to enhance readability and help in quick finding of the necessary modelling results. This has to be entered as an alphanumeric string (up to 26 characters, no spaces).
